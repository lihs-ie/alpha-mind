# 運用設計

最終更新日: 2026-02-28

## 1. 目的

- 本書は、AI投資運用アプリ（イベント駆動マイクロサービス）の外部運用設計を定義する。
- 単一ユーザー運用（想定ユーザー数: 1）で、低コストかつ安全性を優先した運用標準を定める。
- 内部実装ではなく、運用者から見える運用手順・基準・責任分界を記述する。

## 2. 適用範囲

- 対象システム: `bff`, `data-collector`, `feature-engineering`, `signal-generator`, `portfolio-planner`, `risk-guard`, `execution`, `audit-log`, `insight-collector`, `agent-orchestrator`, `hypothesis-lab`
- 関連仕様:
  - `外部設計/api/openapi.yaml`
  - `外部設計/api/asyncapi.yaml`
  - `外部設計/state/状態遷移設計.md`
  - `外部設計/error/error-codes.json`
  - `外部設計/security/認証認可設計.md`

## 3. 運用設計書の書き方（調査結果）

### 3.1 採用した標準・ガイド

1. SRE（Google SRE Book / SRE Workbook）
- 運用の中核を `SLI/SLO`、`Error Budget`、`Burn Rate Alert`、`On-Call`、`Incident Response`、`Postmortem` で統一する。

2. Google Cloud Well-Architected（Operational Excellence）
- Runbook/Playbook、役割定義、継続的改善、問題管理を運用章に必須化する。

3. NIST SP 800-61 Rev.2
- セキュリティインシデント対応を `準備 -> 検知/分析 -> 封じ込め/根絶/復旧 -> 事後対応` で設計する。

4. Google Cloud DR Planning Guide + Firestore DR Guidance
- `RTO/RPO` を明示し、復旧手順を定量目標で管理する。

## 4. 運用モデル

### 4.1 体制（MVP）

- 運用責任者: 1名（開発者兼運用者）
- 平時: 日次バッチ監視 + 週次レビュー
- 障害時: 同一人物が Incident Commander として初動し、Runbookに従い復旧する

### 4.2 責務分界

- BFF: 認証、認可、UI向け集約、運用コマンド受付
- イベント系サービス: 収集/特徴量/シグナル/計画/リスク/執行
- 監査: すべての意思決定・手動操作の記録
- 運用者: kill switch、承認/拒否、再実行、モデル昇格判定（個別株は手動、ETFは条件付き自動）

## 5. 監視設計

### 5.1 監視対象

- 可用性: API成功率、イベント処理成功率
- レイテンシ: BFF応答時間、注文送信から約定確認までの遅延
- 品質: データ鮮度、欠損率、モデル検証成績、インサイト根拠欠損率、仮説重複率
- 品質: 直近期間の性能劣化（Sharpe低下、コスト比率上昇、回転率過多）
- 安全: kill switch状態、リスク上限超過件数
- コスト: Firestore read/write/delete、Cloud Run実行時間、外部API呼び出し回数

### 5.2 ログ・トレース標準

- すべてのリクエスト/イベントで `trace` を必須化
- 構造化ログに `service`, `eventType`, `reasonCode`, `actionReasonCode`, `user`, `identifier` を含める
- 監査ログは `7年` 保持（規制/検証証跡目的）
- Skill実行ログは `skill` `skillVersion` `instructionProfileVersion` を必須記録

## 6. SLI/SLO・Error Budget

### 6.1 SLOカタログ（初期値）

| ID | 対象 | SLI | SLO（30日） | 備考 |
|---|---|---|---|---|
| SLO-001 | BFF 可用性 | 2xx/3xx 比率 | 99.5%以上 | `/healthz` 除外 |
| SLO-002 | 日次データ収集 | 07:00 JST までの完了率 | 99.0%以上 | 営業日ベース |
| SLO-003 | シグナル生成 | `features.generated` 受信後15分以内成功率 | 99.0%以上 | 失敗時は `signal.generation.failed` |
| SLO-004 | 注文実行 | 承認済み注文の送信成功率 | 99.5%以上 | kill switch有効時は対象外 |
| SLO-005 | kill switch 反映 | 有効化後30秒以内に新規発注停止 | 99.9%以上 | 安全性最優先 |
| SLO-006 | インサイト収集品質 | 根拠リンク付きインサイト比率 | 99.0%以上 | `sourceUrl` 欠損禁止 |
| SLO-007 | 仮説検証完了率 | 提案仮説の7日以内検証完了率 | 95.0%以上 | 放置仮説を防止 |

### 6.2 Error Budgetポリシー

- 予算消費 < 50%: 通常運用
- 予算消費 50%以上: 新機能リリース凍結、信頼性改善を優先
- 予算消費 100%以上: 変更停止、障害要因の是正完了まで復旧作業に集中

## 7. アラート設計（Burn Rate）

### 7.1 方針

- アラートは閾値超過ではなく「Error Budget消費速度（Burn Rate）」で評価する。
- 重大通知はマルチウィンドウ（短期+長期）で誤検知を抑制する。

### 7.2 初期ルール（30日SLO）

1. `critical-page`
- 条件: `burn rate >= 14.4` を「5分 AND 1時間」で同時検知
- 目的: 大規模障害の早期検知

2. `high-page`
- 条件: `burn rate >= 6` を「30分 AND 6時間」で同時検知
- 目的: 継続劣化の検知

3. `ticket`
- 条件: `burn rate >= 1` を「6時間 AND 3日」で検知
- 目的: 慢性的劣化の改善タスク化

## 8. インシデント対応

### 8.1 重大度定義

| Severity | 影響 | 例 | 初動目標 |
|---|---|---|---|
| SEV1 | 取引安全性に重大影響 | kill switch不作動、誤発注連鎖 | 5分以内に宣言 |
| SEV2 | 主要機能停止 | 日次パイプライン停止、注文不能 | 15分以内に対応開始 |
| SEV3 | 一部機能劣化 | ダッシュボード遅延、再試行で回復 | 営業時間内対応 |
| SEV4 | 軽微/改善課題 | 表示崩れ、軽微な警告増加 | 次回スプリント対応 |

### 8.2 標準フロー

1. 検知: Alert受信、影響確認
2. 宣言: Severity確定、`trace`/チケット発行
3. 封じ込め: kill switch、該当サービス停止、ロールバック
4. 復旧: Runbookに従い段階復旧
5. 事後: 24時間以内にポストモーテム作成

### 8.3 連絡・記録

- 失敗判断は `外部設計/error/error-codes.json` の `reasonCode`、運用者/自動判定理由は `actionReasonCode` で分類
- 監査ログに「時刻、判断者、判断理由、影響範囲、復旧時刻」を必須記録

## 10. 変更管理（リリース/ロールバック）

### 10.1 リリース方針

- 変更は原則 `小さく・頻繁に`。一度に複数サービスを同時変更しない。
- BFFは段階配信（canary）を前提とし、失敗時は即時ロールバック可能な構成とする。

### 10.2 リリースゲート

1. `pnpm type-check` 成功
2. OpenAPI/AsyncAPI差分レビュー完了
3. 主要SLOが直近7日で安定
4. 監査ログ項目に欠損なし

### 10.3 ロールバック条件

- `SLO-001` か `SLO-004` が急落
- SEV1/SEV2インシデント発生
- 認証/認可の不整合を検知

## 11. バックアップ/DR設計

### 11.1 目標値

| データ種別 | RTO | RPO | 復旧手段 |
|---|---|---|---|
| Firestore（設定・状態・監査） | 15分 | 1分 | PITR / バックアップ復元 |
| Cloud Storage（履歴データ） | 4時間 | 24時間 | Object Versioning + 再取込 |
| 一時イベントデータ | 24時間 | 24時間 | 再実行で再生成 |

### 11.2 実装方針

- Firestore: PITR有効化、定期バックアップ、四半期ごとに復元訓練
- Cloud Storage: Object Versioning + Lifecycleで世代管理
- Pub/Sub: DLQを必須化し、最大配信回数超過メッセージを隔離

## 12. セキュリティ運用

- 認証鍵: Secret Manager管理、四半期ローテーション
- 権限: 最小権限（IAM）を維持し、月次棚卸し
- 監査: 認可拒否率（401/403）急増をアラート対象化
- インシデント対応は NIST SP 800-61 の4段階に沿って実施

## 13. コスト運用（単一ユーザー最適化）

- Cloud Runはデフォルトで `min instances=0`（スケールtoゼロ）を維持
- 常時稼働が不要な処理は Cloud Scheduler + Job 実行に集約
- Firestoreは read最適化を優先し、Listener常時接続を避ける
- 月次の運用費しきい値: `60 USD`。超過見込み時は運用頻度/対象銘柄数を縮小

## 14. 運用レビュー（継続改善）

### 14.1 週次

- SLO達成率、アラート件数、誤検知率、手動介入回数を確認
- Skill品質レビュー（精度、失敗率、実行コスト、利用規約違反検知件数）を実施し、不要Skillを棚卸し
- Runbookの更新要否を判定

### 14.2 月次

- Error Budget消費分析
- コスト実績（Firestore/Cloud Run/外部データAPI）と予算差異分析
- モデル運用の安全性指標（最大DD、回転率、拒否率）確認

### 14.3 四半期

- DR訓練（復元リハーサル）
- 鍵ローテーション監査
- ポストモーテムの再発防止策の完了確認

## 15. 受入基準（Definition of Done）

1. すべてのSLOに計測クエリとアラートルールが定義されている
2. SEV1/SEV2のRunbookが整備され、演習で実行確認済み
3. Firestore復元手順が検証済み（RTO/RPO達成）
4. kill switch動作確認が自動テストで担保されている
5. 運用レビュー（週次/月次/四半期）の定型テンプレートが存在する
6. AI提案戦略に対するコスト控除評価とコンプライアンス確認のチェックリストが運用されている
7. 失敗知見DBが運用され、重複仮説抑止が確認できる
