# 監視クエリ設計（SLO/Alert）

最終更新日: 2026-03-03

## 1. 目的

- `外部設計/operations/運用設計.md` で定義したSLOを、Cloud Monitoring上で実装できる計測クエリとして具体化する。
- `外部設計/operations/slo-catalog.json` の論理定義を、メトリクス/フィルタ/ウィンドウに落とし込む。

## 2. 適用範囲

- 対象SLO: `SLO-001` 〜 `SLO-011`
- 対象補助モニタ: `MON-001` 〜 `MON-004`
- 監視基盤: Google Cloud Monitoring（SLO + Alerting）
- 対象サービス: `bff`, `data-collector`, `feature-engineering`, `signal-generator`, `portfolio-planner`, `risk-guard`, `execution`, `audit-log`, `insight-collector`, `agent-orchestrator`, `hypothesis-lab`

## 3. 計測方式

### 3.1 原則

1. API可用性はマネージドメトリクスを使用する。
2. 業務SLO（締切順守、15分以内成功など）はカスタムメトリクスを使用する。
3. SLIは必ず `numerator / denominator` で定義する。
4. Burn Rateは `error_ratio / allowed_error_ratio` で評価する。

### 3.2 共通式

- `sli_ratio = numerator / denominator`
- `error_ratio = 1 - sli_ratio`
- `allowed_error_ratio = 1 - slo_objective`
- `burn_rate = error_ratio / allowed_error_ratio`

## 5. SLO別クエリ仕様

## 5.1 SLO-001 BFF可用性（99.5%）

- 分母（total requests）
  - metric: `run.googleapis.com/request_count`
  - filter:
    - `resource.type="cloud_run_revision"`
    - `resource.labels.service_name="bff"`
    - `metric.labels.response_code_class!~"4xx|5xx"` は使用しない（分母は全件）
    - `metric.labels.route!="/healthz"`（ヘルスチェック除外。実装時はラベル仕様に合わせて path/route を選択）
- 分子（good requests）
  - 上記に加えて `metric.labels.response_code_class=~"2xx|3xx"`
- 集計:
  - 1分粒度で合計し、30日窓で比率計算

## 5.2 SLO-002 日次データ収集締切順守（99.0%）

- 分母: `custom.googleapis.com/invest_ai/data_collection/runs_total`
- 分子: `custom.googleapis.com/invest_ai/data_collection/runs_on_time_total`
- ラベル:
  - `market`（jp/us）, `schedule`（daily/weekly）, `date`
- 集計:
  - 営業日単位で1日1サンプルを確定
  - 30日窓で比率計算

## 5.3 SLO-003 15分以内シグナル生成成功率（99.0%）

- 分母: `custom.googleapis.com/invest_ai/signal/features_generated_total`
- 分子: `custom.googleapis.com/invest_ai/signal/generated_within_15m_total`
- ラベル:
  - `model_version`, `market`, `pipeline_id`
- 集計:
  - イベント単位で比率計算

## 5.4 SLO-004 承認済み注文の送信成功率（99.5%）

- 分母: `custom.googleapis.com/invest_ai/execution/approved_orders_total`
- 分子: `custom.googleapis.com/invest_ai/execution/orders_sent_success_total`
- ラベル:
  - `market`, `broker`, `order_type`
- 除外条件:
  - `kill_switch_state="enabled"` のイベントは分母に含めない

## 5.5 SLO-005 kill switch反映（99.9%）

## 5.6 SLO-006 インサイト根拠リンク付与率（99.0%）

- 分子: `custom.googleapis.com/invest_ai/insight/with_evidence_total`

## 5.7 SLO-007 仮説7日以内検証完了率（95.0%）

## 6. Burn Rateアラート仕様

| Alert ID | 条件 | ウィンドウ | 通知レベル | 対応 |
|---|---|---|---|---|
| ALERT-CRITICAL | `burn_rate >= 14.4` | 5m AND 1h | page | SEV1/SEV2判定、即時封じ込め |
| ALERT-HIGH | `burn_rate >= 6` | 30m AND 6h | page | 原因調査、必要時ロールバック |
| ALERT-TICKET | `burn_rate >= 1` | 6h AND 3d | ticket | 改善タスク化、次スプリント反映 |

## 7. 実装ポリシー（IaC前提）

1. SLOの正本は `外部設計/operations/slo-catalog.json`。
2. クエリ実装の正本は `外部設計/operations/slo-query-spec.json`。
3. 実環境反映はTerraform等で自動化し、手動設定を残さない。
4. アラート通知先は `page` と `ticket` を分離する。
5. 補助モニタ（`supplementalMonitors`）はSLOとは別ポリシーで管理する。

## 8. データ品質ガード

- 分母が0の窓はSLO計算対象外にする（NoDataで誤判定しない）。
- カスタムメトリクス送信遅延（>5分）を別アラート化する。
- `trace` 欠損イベントは監査ログに `DATA_VALIDATION_FAILED` で記録する。

## 9. 受入基準

1. 11個のSLOすべてで numerator/denominator が実装されている。
2. 3段階のBurn Rateアラートが設定済みである。
3. NoData時の扱いが明文化され、誤検知が抑制されている。
4. 監視設定変更はPull Requestでレビューされる。
5. `MON-001`〜`MON-004` の補助モニタが実装され、Runbookに接続されている。
