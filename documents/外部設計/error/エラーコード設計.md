# エラーコード設計

最終更新日: 2026-02-28

## 1. 目的

- BFF API（OpenAPI）とイベント契約（AsyncAPI）で、共通のエラー表現を定義する。
- エラーの原因・再試行可否・対応主体を明確化し、運用判断を高速化する。

## 2. 適用範囲

- 同期API: `外部設計/api/openapi.yaml`
- 非同期イベント: `外部設計/api/asyncapi.yaml`
- 対象サービス: `bff` と各マイクロサービス

## 3. 基本方針

1. すべての失敗は `reasonCode` で分類する。  
2. APIエラーは `application/problem+json` で返す。  
3. イベント失敗は `*.failed` と `payload.reasonCode` で表現する。  
4. 再試行可否は `retryable` で明示する。  
5. すべてのエラーに `trace` を含める。  

## 4. エラーモデル

### 4.1 API（Problem Details）

レスポンス例:

```json
{
  "type": "about:blank",
  "title": "Conflict",
  "status": 409,
  "detail": "kill switch is enabled",
  "instance": "/orders/123/approve",
  "trace": "trc_...",
  "reasonCode": "KILL_SWITCH_ENABLED",
  "retryable": false
}
```

### 4.2 Event（Failed Payload）

```json
{
  "identifier": "...",
  "eventType": "orders.execution.failed",
  "occurredAt": "2026-02-12T09:00:00Z",
  "trace": "trc_...",
  "schemaVersion": "1.0.0",
  "payload": {
    "reasonCode": "EXECUTION_BROKER_TIMEOUT",
    "detail": "broker api timeout"
  }
}
```

## 5. コード体系

命名規則:
- `UPPER_SNAKE_CASE`
- ドメインを先頭に付与（例: `AUTH_*`, `DATA_*`, `EXECUTION_*`）

カテゴリ:
- `AUTH`: 認証・認可
- `CLIENT`: 入力/参照エラー
- `BUSINESS`: 業務ルール違反
- `DATA`: データ品質・スキーマ
- `DEPENDENCY`: 外部/内部依存障害
- `SYSTEM`: 予期しない内部障害

## 6. エラーコード一覧

機械可読の正本:
- `外部設計/error/error-codes.json`

主要コード（抜粋）:
- `AUTH_INVALID_CREDENTIALS` (401, retryable=false)
- `REQUEST_VALIDATION_FAILED` (400, retryable=false)
- `STATE_CONFLICT` (409, retryable=false)
- `KILL_SWITCH_ENABLED` (409, retryable=false)
- `RISK_LIMIT_EXCEEDED` (409, retryable=false)
- `RISK_EVALUATION_UNAVAILABLE` (409, retryable=false)
- `MODEL_NOT_APPROVED` (409, retryable=false)
- `COMPLIANCE_REVIEW_REQUIRED` (409, retryable=false)
- `COMPLIANCE_RESTRICTED_SYMBOL` (409, retryable=false)
- `COMPLIANCE_BLACKOUT_ACTIVE` (409, retryable=false)
- `COMPLIANCE_MNPI_SUSPECTED` (422, retryable=false)
- `COMPLIANCE_SOURCE_UNAPPROVED` (422, retryable=false)
- `DATA_SOURCE_TIMEOUT` (503, retryable=true)
- `DATA_QUALITY_LEAK_DETECTED` (422, retryable=false)
- `EXECUTION_BROKER_TIMEOUT` (503, retryable=true)
- `EXECUTION_INSUFFICIENT_FUNDS` (409, retryable=false)
- `INTERNAL_ERROR` (500, retryable=true)

## 7. HTTPステータス対応方針

| HTTP | 用途 | 代表コード |
|---|---|---|
| 400 | 入力バリデーション不正 | `REQUEST_VALIDATION_FAILED` |
| 401 | 認証失敗 | `AUTH_INVALID_CREDENTIALS`, `AUTH_TOKEN_EXPIRED` |
| 403 | 認可失敗 | `AUTH_FORBIDDEN` |
| 404 | 未存在 | `RESOURCE_NOT_FOUND`, `MODEL_NOT_FOUND` |
| 409 | 状態競合/業務制約 | `STATE_CONFLICT`, `KILL_SWITCH_ENABLED`, `RISK_LIMIT_EXCEEDED`, `COMPLIANCE_RESTRICTED_SYMBOL` |
| 422 | データ品質/コンプライアンス異常 | `DATA_SCHEMA_INVALID`, `DATA_QUALITY_LEAK_DETECTED`, `COMPLIANCE_MNPI_SUSPECTED` |
| 500 | システム障害 | `INTERNAL_ERROR`, `FEATURE_GENERATION_FAILED` |
| 503 | 依存先障害 | `DEPENDENCY_TIMEOUT`, `EXECUTION_BROKER_TIMEOUT` |

## 8. 再試行方針

1. `retryable=true`
- クライアント/BFF: 最大3回（指数バックオフ）
- 非同期処理: 既存のPub/Sub再試行 + DLQ

2. `retryable=false`
- 即時失敗として返す
- 利用者操作（設定変更、時間変更、承認見直し）を促す

## 9. 監査・運用

- API失敗とイベント失敗の両方で `reasonCode` を監査ログに記録する。
- 監視ダッシュボードは `reasonCode` 別件数を表示する。
- アラートは `SYSTEM` / `DEPENDENCY` カテゴリで集約する。

## 10. 変更管理

- `error-codes.json` の追加は後方互換（non-breaking）。
- 既存コード削除・意味変更は breaking change とする。
- OpenAPI/AsyncAPIの `ReasonCode` enum と同時更新を必須化する。
