# execution 外部設計書

最終更新日: 2026-02-28

## 1. サービス概要

- 役割: 承認済み注文をブローカーへ送信する執行サービス。
- 主な責務: 発注、約定状態同期、執行結果イベント発行。
- 主な利用者: `audit-log`、Webコンソール。

## 2. 採用技術と比較

| 項目 | 採用 | 比較対象 | 採用理由 |
|---|---|---|---|
| ブローカー連携 | Alpaca API（MVP） | 複数ブローカー同時対応 | 初期統合対象を絞って外部契約を安定化 |
| 実行基盤 | Cloud Run | 常駐VM | 低頻度執行で固定費を抑制 |
| 状態保存 | Firestore | ローカルファイル | UI参照と監査連携を容易にする |

## 3. 外部インターフェース

### 3.1 購読イベント

- `orders.approved`

### 3.2 発行イベント

- `orders.executed`
- `orders.execution.failed`
- `hypothesis.demo.completed`（デモ運用モード時）

### 3.3 外部依存

- ブローカー注文API
- Firestore（注文状態、約定履歴、demo_trade_runs）

## 4. ユースケース

### UC-EX-01: 注文執行

- 事前条件: `orders.approved`が受信済み。
- トリガー: 執行要求イベント。
- 成果: 発注結果を保存し、`orders.executed`発行。

### UC-EX-02: 外部API障害

- 事前条件: ブローカーAPIが一時的に失敗。
- トリガー: 発注時HTTPエラー。
- 成果: 再試行上限到達時に`orders.execution.failed`発行。

### UC-EX-03: デモ期間完了通知

- 事前条件: 仮説が`demo`状態で評価期間を満たす。
- トリガー: デモ評価ジョブ完了。
- 成果: `hypothesis.demo.completed` を発行し、昇格判定に必要な指標を連携する。

## 5. 非機能要件

- 執行遅延目標: 承認イベント受信から`30秒以内`。
- 冪等性: 同一`identifier`の重複執行を禁止。
- 可観測性: 注文成功率、失敗率、外部APIレイテンシを監視。

## 6. スコープ外

- ブローカー選定ロジック。
- リスク判定。
- 戦略評価。
