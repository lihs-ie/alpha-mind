# feature-engineering 外部設計書

最終更新日: 2026-02-27

## 1. サービス概要

- 役割: 学習・推論用特徴量の生成サービス。
- 主な責務: 取得済み市場データと定性インサイトを統合し、特徴量を生成してバージョン付きで保存する。
- 主な利用者: `signal-generator`。

## 2. 採用技術と比較

| 項目 | 採用 | 比較対象 | 採用理由 |
|---|---|---|---|
| 実行基盤 | Cloud Run Job | 常駐Worker | 日次/週次処理中心で常駐不要 |
| 保存先 | Cloud Storage(Parquet) | Firestore | 列指向形式で再学習・再現性管理に適する |
| 契約 | CloudEvents互換JSON | 独自イベント | イベント契約の可読性と一貫性を維持 |

## 3. 外部インターフェース

### 3.1 購読イベント

- `market.collected`

### 3.2 発行イベント

- `features.generated`
- `features.generation.failed`

### 3.3 外部依存

- Cloud Storage（入力データ/出力特徴量）
- Firestore（処理状態メタ情報、insight_records）

## 4. ユースケース

### UC-FE-01: 特徴量生成

- 事前条件: 対象日の市場データが保存済み。
- トリガー: `market.collected`受信。
- 成果: 定量・定性融合特徴量ファイルを保存し、`features.generated`発行。

### UC-FE-03: 定性×定量特徴量の融合

- 事前条件: 対象期間のインサイトが `insight_records` に保存済み。
- トリガー: `market.collected`受信。
- 成果: 逆日歩、価格系列、インサイトセンチメントを同一時点で結合した特徴量を生成。

### UC-FE-02: 入力不整合時の停止

- 事前条件: 入力データに必須列欠損。
- トリガー: バリデーション失敗。
- 成果: `features.generation.failed`を発行し後続処理を抑止。

## 5. 非機能要件

- 完了目標: 1サイクル`15分以内`。
- データ品質: 将来情報の混入を検知した場合は失敗扱い。
- データ品質: 定性データは `collectedAt <= targetDate` を満たすもののみ利用する。
- 監査: 特徴量バージョンと`trace`を必ず記録。

## 6. スコープ外

- モデル学習。
- シグナル生成ロジック。
- 発注判断。
