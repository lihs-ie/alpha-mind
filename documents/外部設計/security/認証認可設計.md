# 認証・認可設計

最終更新日: 2026-02-28

## 1. 目的

- BFF中心アーキテクチャにおける認証（Authentication）と認可（Authorization）の仕様を定義する。
- 単一ユーザーMVPで必要な安全性・監査性を確保し、将来の多ユーザー化に備える。

## 2. 適用範囲

- 公開API: `bff`
- 内部サービス: `data-collector` 〜 `audit-log`
- 対象仕様: `外部設計/api/openapi.yaml`, `外部設計/api/asyncapi.yaml`

## 3. 全体方針

1. 外部公開はBFFのみ。内部サービスはインターネット非公開。  
2. API認証はBearer JWT（OIDC準拠）を使用。  
3. 認可は「ロール + 権限（permission）」で判定。  
4. すべての更新系操作に `trace` と `user` を付与して監査する。  
5. 認証・認可失敗は `reasonCode` 付き Problem Details で返す。  

## 4. 認証設計

### 4.1 ログイン

- エンドポイント: `POST /auth/login`
- 入力: `email`, `password`
- 出力: `accessToken`（JWT）, `tokenType`, `expiresIn`, `user`

### 4.2 トークン仕様

- 形式: JWT
- 署名アルゴリズム: RS256（推奨）
- 伝達: `Authorization: Bearer <token>`
- 有効期限: 60分（MVP基準）
- 許容時刻ずれ: 60秒

### 4.3 必須クレーム

| クレーム | 必須 | 説明 |
|---|---|---|
| `iss` | 必須 | 発行者 |
| `aud` | 必須 | API Audience |
| `sub` | 必須 | ユーザー識別子 |
| `email` | 必須 | ユーザーメール |
| `role` | 必須 | `admin` / `viewer` |
| `permissions` | 必須 | 権限文字列配列 |
| `iat` | 必須 | 発行時刻 |
| `exp` | 必須 | 失効時刻 |
| `jti` | 必須 | トークン識別子 |

## 5. 認可設計

### 5.1 ロールモデル

- `admin`: MVP運用者。すべての操作を許可。
- `viewer`: 将来拡張用の参照専用ロール。

### 5.2 権限マトリクス

正本:
- `外部設計/security/authz-matrix.json`

主要権限:
- `dashboard:read`
- `operations:write`
- `commands:run`
- `insights:read`, `insights:write`
- `hypotheses:read`
- `hypotheses:retest`
- `hypotheses:decide`
- `settings:read`, `settings:write`
- `compliance:read`, `compliance:write`
- `orders:read`, `orders:approve`, `orders:reject`, `orders:retry`
- `audit:read`
- `models:read`, `models:decide`

### 5.3 判定順序（BFF）

1. JWT署名・有効期限検証
2. 必須クレーム検証
3. 対象エンドポイントの required permission 解決
4. `permissions` との照合
5. 可否結果を監査ログへ記録

## 6. サービス間認可（内部通信）

- 原則: ユーザートークンで内部サービスを直接呼ばない。
- サービス間通信は GCP IAM（Service Account）で認可。
- Pub/Sub publish/subscribe 権限は最小権限で付与する。
- Firestoreは Admin SDK + IAM でアクセスし、クライアント直接アクセスは禁止。

## 6.1 インサイダー接触回避の技術制約

1. データソース制約
- 許可ソース分類（公開済み市場データ、公開開示情報）以外は取り込み禁止。
- 許可ソース分類は `compliance_controls.sourcePolicies` と `source_policies` コレクションを正本として判定する。
- `publishedAt` が確認できないデータは `COMPLIANCE_SOURCE_UNAPPROVED` で拒否する。

2. 手動入力制約
- 発注・モデル昇格・緊急操作の入力は「コード化理由 + 短文コメント」に限定する。
- コメントは最大120文字、改行不可、MNPI疑義フィルタを通過した場合のみ受け付ける。
- MNPI疑義時は `COMPLIANCE_MNPI_SUSPECTED` を返し、処理を中断する。
- 昇格操作では `MNPIを知らない` 自己申告を必須入力とし、監査ログに保存する。

3. 取引実行制約
- `restrictedSymbols`（制限銘柄）に該当する注文は常に拒否する。
- `partnerRestrictedSymbols`（取引先・案件関連銘柄）に該当する注文/昇格は常に拒否する。
- `blackoutWindows`（ブラックアウト期間）中の対象銘柄注文は常に拒否する。
- 拒否時は `COMPLIANCE_RESTRICTED_SYMBOL` / `COMPLIANCE_BLACKOUT_ACTIVE` を返す。

4. モデル昇格制約
- `requiresComplianceReview=true` のモデルは昇格不可。
- 昇格要求は `COMPLIANCE_REVIEW_REQUIRED` で拒否する。

5. 監査制約
- 上記制約の判定結果を `trace`, `user`, `symbol`, `reasonCode` で監査ログに記録する。
- 判定イベントは運用Runbook（`RB-AI-002`）に連動させる。

### 6.2 自動昇格のインサイダーリスク評価

結論:
- 「無条件の自動昇格」は不許可。
- 「条件付き自動昇格」は許可可能（ETF中心、低インサイダーリスク条件付き）。

評価観点:
1. 現行で有効な対策
- ソース制約（公開情報のみ）
- MNPI疑義フィルタ（コメント入力）
- `requiresComplianceReview` ガード
- `trace` と監査ログによる事後追跡

2. 自動昇格時の残余リスク
- 誤検知/見逃し（false negative）を前提にした二次防御が弱くなる。
- 「未公表重要事実を知った後の取引」に該当するおそれがある局面で、実行前の人手判断が介在しない。
- 事後監査で説明可能でも、実行前抑止としては不十分。

3. 設計上の必須制約
- 自動昇格の対象は `instrumentType=ETF` かつ `insiderRisk=low` の仮説に限定する。
- 個別株（`instrumentType=STOCK`）は `POST /hypotheses/{identifier}/promote` の手動承認を必須とする。
- 昇格時は `MNPIを知らない` 自己申告（`mnpiSelfDeclared=true`）を必須記録する。自己申告が無効/欠損時は自動昇格を禁止する。
- `partnerRestrictedSymbols`（取引先・案件関連銘柄）に該当する場合は自動昇格を禁止する。
- 昇格時は `reasonCode` + `trace` + `user` を必須監査項目とする。
- `demoPeriodDays>=30` と `requiresComplianceReview=false` を同時に満たさない場合は拒否（`COMPLIANCE_REVIEW_REQUIRED` or `OPERATION_NOT_ALLOWED`）。

詳細設計:
- `外部設計/security/条件付き自動昇格設計.md`

## 7. エラー設計

### 7.1 APIエラー

- 形式: `application/problem+json`
- 主要コード:
- `AUTH_INVALID_CREDENTIALS` (401)
- `AUTH_TOKEN_EXPIRED` (401)
- `AUTH_FORBIDDEN` (403)
- `REQUEST_VALIDATION_FAILED` (400)

### 7.2 認可拒否時のレスポンス例

```json
{
  "type": "about:blank",
  "title": "Forbidden",
  "status": 403,
  "detail": "required permission: orders:approve",
  "trace": "trc_xxx",
  "reasonCode": "AUTH_FORBIDDEN",
  "retryable": false
}
```

## 8. 監査要件

認証・認可イベントで記録する必須項目:
- `timestamp`
- `trace`
- `user` (`sub`)
- `email`
- `role`
- `permission`
- `endpoint` (`method`, `path`)
- `result` (`allow` / `deny`)
- `reasonCode`

## 9. セキュリティ運用

1. トークン鍵管理
- 署名鍵はSecret Manager保管
- 鍵ローテーションは四半期ごと

2. 侵害対策
- ログイン失敗回数のレート制限
- 異常な401/403増加でアラート

3. ネットワーク
- BFFはHTTPSのみ許可
- 内部サービスはPrivateアクセスのみ

## 10. テスト観点

1. 正常系
- 正しいJWTで許可される

2. 異常系
- 期限切れトークンで401
- 権限不足で403
- 改ざんトークンで401

3. 監査
- deny時に `reasonCode` と `trace` が記録される

## 11. 変更管理

- 権限追加・削除は `authz-matrix.json` を正本として管理する。
- OpenAPIに新規エンドポイント追加時は、同時に権限マトリクス更新を必須とする。
