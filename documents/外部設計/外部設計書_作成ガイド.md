# 外部設計書 作成ガイド（イベント駆動マイクロサービス）

最終更新日: 2026-02-24

## 1. このガイドの目的

- 本ガイドは、AI投資運用アプリの外部設計書をマイクロサービス単位で作成するための共通ルールを定義する。
- ここでの「外部設計」は、利用者・連携先・運用者から見える仕様を定義する文書であり、内部アルゴリズムやクラス設計は対象外とする。

## 2. 調査結果（外部設計書の書き方）

外部設計として有効だった標準・実務フレームワークは以下。

1. ISO/IEC/IEEE 42010
- 目的: アーキテクチャ記述を「利害関係者・関心事・ビュー」で整理する。
- 外部設計への使い方: 誰向けの文書かを最初に明記し、ビュー（業務、連携、運用）を分離する。

2. arc42
- 目的: 実務で使えるアーキテクチャ文書テンプレート。
- 外部設計への使い方: コンテキスト、制約、品質、ランタイム、デプロイを章立てとして利用する。

3. C4 Model
- 目的: システム境界と責務を段階的に可視化する（Context/Container/Component/Code）。
- 外部設計への使い方: Context図とContainer図までを必須化し、Component以下は内部設計で管理する。

4. OpenAPI / AsyncAPI / CloudEvents
- 目的: 同期API・非同期イベント契約を機械可読で定義する。
- 外部設計への使い方: RESTはOpenAPI、イベントはAsyncAPIまたはCloudEvents準拠JSONで仕様化する。

## 3. 外部設計書の原則

1. 1サービス1文書
- 例: `signal-generator.md`。

2. 契約駆動
- 実装前に「入出力契約（HTTP/Event）」を先に確定する。

3. 内部実装を書かない
- 何をするか（責務）と、どう見えるか（契約・品質）だけを書く。

4. 変更可能性を明示
- 将来変更が起きる点（データ提供元、注文先、モデル）を「設計判断」として記録する。

## 4. マイクロサービス外部設計書テンプレート

各サービスは必ず次の章を持つ。

1. サービス概要
- 役割（1文）
- 対象アクター（誰が使うか）
- 管理する業務イベント

2. 採用技術と比較
- 採用技術
- 比較対象
- 採用理由（運用コスト、学習コスト、保守性）
- 非採用理由

3. 外部インターフェース
- 同期IF: エンドポイント、認証方式、主要レスポンスコード
- 非同期IF: 購読イベント、発行イベント、再試行/DLQ方針
- 外部依存: DB、ストレージ、外部API

4. ユースケース
- 前提条件
- 基本フロー（業務視点）
- 代替フロー（失敗時、停止時、欠損時）

5. 非機能要件
- 可用性/SLO
- 性能目標（最大遅延、最大処理時間）
- セキュリティ（認証、認可、秘密情報管理）
- 監査（trace、監査ログ保持）

6. 運用要件
- 監視項目
- アラート条件
- 手動介入手順（kill switch、再実行）

7. スコープ外
- このサービスが担当しないことを明記する。

## 5. 技術選定の記載方法（何を書くべきか）

### 5.1 記載ルール

- 「採用技術名」だけでなく「比較対象」と「採用理由」を必ずセットで書く。
- 理由は次の3軸で評価する。
- コスト: 固定費/従量課金、運用人数、監視負荷
- リスク: 過学習、データリーク、障害時影響
- 将来性: 市場追加・ブローカー追加時の拡張性

### 5.2 本プロジェクトの推奨スタック（外部設計観点）

| 領域 | 推奨 | 比較対象 | 外部設計での優位点 |
|---|---|---|---|
| 実行基盤 | Cloud Run | GKE, Functions | サービスごとのデプロイ境界が明確で、`min instances=0`により低コスト運用しやすい |
| イベント基盤 | Pub/Sub | Kafka, RabbitMQ | 再試行/DLQをマネージドで扱え、運用要件を仕様化しやすい |
| DB | Firestore | PostgreSQL(Cloud SQL/Supabase) | 単一ユーザーMVPで固定費を抑え、サービス単位の責務分離を保ちやすい |
| データ保管 | Cloud Storage(Parquet) | BigQuery主系 | 履歴保存コストを抑え、再学習入力をバージョン管理しやすい |
| 同期契約 | OpenAPI | 独自仕様 | API仕様を共通フォーマットで合意できる |
| 非同期契約 | AsyncAPI/CloudEvents | 独自JSON | 発行/購読契約と互換性方針を文書化しやすい |

## 6. ユースケース記述ルール

- 1ユースケースにつき「1トリガー + 1業務成果」を原則とする。
- 各ユースケースは次の形式で書く。

| 項目 | 記載内容 |
|---|---|
| UC ID | 例: UC-SG-01 |
| 名称 | 例: 日次シグナル生成 |
| 主体 | 自動バッチ / 運用者 |
| 事前条件 | モデルが`approved`状態 |
| トリガー | `features.generated`イベント受信 |
| 基本フロー | 入力検証→スコア生成→`signal.generated`発行 |
| 代替フロー | 入力欠損時は`signal.generation.failed`発行 |
| 事後条件 | 監査ログにtrace付きで記録 |

## 7. 研究結果を外部設計へ反映する書き方

研究で示された課題を、各文書の「非機能要件」と「運用要件」に必ず落とし込む。

| 研究課題 | 外部設計での記述先 | 記述例 |
|---|---|---|
| コスト侵食 | ユースケース / 非機能 | 売買回転率の上限と発注抑制条件 |
| 過学習 | 運用要件 | モデル昇格条件（Walk-forward, DSR/PBO） |
| シグナル劣化 | 監視要件 | 直近成績悪化で自動縮小運用 |
| レジーム変化 | 代替フロー | 異常ボラ時は注文拒否・通知 |
| 分析ノウハウの散逸 | データ設計 / 運用要件 | Skill台帳、失敗知見DB、指示書プロファイルを永続管理 |

## 8. 設計レビュー観点（チェックリスト）

- サービス責務が1つに定義されているか。
- 同期APIとイベント契約が両方明記されているか。
- 認証/認可/監査項目が仕様化されているか。
- 障害時フロー（再試行、DLQ、手動復旧）が記載されているか。
- SLO/アラート条件が数値で定義されているか。
- スコープ外が明記され、過剰責務を避けているか。
